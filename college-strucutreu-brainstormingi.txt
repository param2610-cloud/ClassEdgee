  ---
  The Subject/Course Architecture

  This is the trickiest part. The schema has: courses → syllabus_structure →
  subject_details → units.

  Interpretation:
  - courses = an academic subject paper (e.g. "Data Structures", "Engineering
  Physics")
  - syllabus_structure = "this subject lives in semester X of department Y"
  - subject_details = the detailed spec (theory/lab/project, name, code,
  preferred specializations)
  - units = 5 units with hours + learning objectives
  - faculty_subject_mapping = which faculty can teach which subject_details

  The shared-subject problem:

  Sem 1 & 2 are common to ALL B.Tech branches at MAKAUT. Engineering Mathematics
   I is taught to CSE, ECE, Civil, Mech, Auto simultaneously — same subject,
  same content, but different sections.

  Two valid approaches:

  Option A — One course per subject (shared)
    "Engineering Mathematics I" → course_id=1
    syllabus_structure: course_id=1, semester=1 (shared)
    All departments' sections point to same subject
    PRO: no duplication  |  CON: faculty mapping gets complex

  Option B — One course per subject per department
    "Engg Math I (CSE)" → course_id=1
    "Engg Math I (ECE)" → course_id=2
    PRO: clean faculty-dept isolation  |  CON: 5× duplication for common
  subjects

  Recommendation: Option A (shared) — assign the subject to the Basic Sciences
  dept, then faculty from BS dept teach all sections. This is exactly how real
  colleges work — the Math dept teaches all branches.

  Layer 7 — Timeslots (The Critical Constraint)

  The schema has @@unique([start_time, end_time, day_of_week]) — so only ONE
  record per time+day combo. This means timeslots are shared across all 
  semesters. The semester field on a timeslot is just metadata (which semester
  "owns" this record for filtering), but the physical slot itself is unique.

  48 timeslots — Mon-Sat (Indian colleges), 8 periods/day:

  Period 2:  09:20 - 10:10
  Period 3:  10:10 - 11:00

  ---
  The Key Design Challenges

  Challenge 1: Timeslot semester filtering
  getTimeSlots filters by semester. But all timeslots share one DB record per
  time+day. Seed sets semester=2 on all timeslots. For the auto-scheduler to
  work across Sem 4, 6, 8 sections too, the fetchData query should remove the
  semester filter from timeslots — since they're physically shared.

  Challenge 2: Subject hours → classes per week
  Each subject has total hours across the semester. The solver needs
  classes_per_week. Formula:
  semester = 18 weeks
  classes_per_week = ceil(total_subject_hours / 18)

  Data Structures (48 hrs) → 3 classes/week
  Workshop Lab (36 hrs)    → 2 sessions/week (but 2-hour blocks)

  Challenge 3: Lab subjects need 2-consecutive-slot blocks
  Labs are typically 2-3 periods back to back. The current solver doesn't
  explicitly handle consecutive slots — it assigns one slot at a time. For labs,
   we either:
  - Model a "lab session" as a single 2-hour timeslot (8:30-10:10 block)
  - Or add consecutive-slot logic to the solver

  Easiest fix: Add 2-hour lab timeslots to the timeslots table (the unique
  constraint allows 8:30-10:10 as a different record from 8:30-9:20).

  Challenge 4: Student enrollment number format
  enrollment_number unique. Use format: SURTECH/YYYY/DEPT/NNN
  Example: SURTECH/2025/CSE/001
